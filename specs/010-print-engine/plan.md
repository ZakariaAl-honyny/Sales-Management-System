# Implementation Plan: Print Engine (v4.3)

**Branch**: `010-print-engine` | **Date**: 2026-05-24 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/010-print-engine/spec.md`

---

## Summary

Implement a two-mode print engine: (1) A4 PDF generation with preview using QuestPDF, and (2) 80mm thermal receipt printing via a hand-built ESC/POS command builder over Win32 raw print — no external NuGet for thermal. All printing is initiated through a dedicated `PrintController` in the API layer. The Desktop never calls print functions directly. All operations return `PrintResult` — no exceptions propagate to callers. Print settings (branding, auto-print) are stored in `SystemSettings` under category `"Print"`.

---

## Technical Context

**Language/Version**: C# 13 / .NET 10 LTS
**Primary Dependencies**:
- A4 PDF: `QuestPDF` (already referenced per RULE-088)
- Thermal: Win32 `winspool.drv` via P/Invoke — NO external NuGet package
- ESC/POS: In-house `EscPosCommandBuilder` class
**Storage**: No new tables. Uses existing `SystemSettings` table (`Category = "Print"`)
**Testing**: xUnit + Moq + FluentAssertions
**Target Platform**: Windows x64 — API as Windows Service
**Performance Goals**: A4 PDF < 3s for 100 items; Thermal job to spooler < 1s
**Constraints**:
- RULE-088: QuestPDF for A4 ONLY, Win32 P/Invoke for thermal, NO WPF FixedDocument
- RULE-006: All service methods return `Result<T>` / `PrintResult` — never throw
- Desktop NEVER contains PDF or print libraries — all via API
- Logo null-check mandatory — missing logo must never crash generation
**Scale/Scope**: 1–5 concurrent print users per store
