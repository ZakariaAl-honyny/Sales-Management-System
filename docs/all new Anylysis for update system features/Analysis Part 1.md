# تحليل تطبيق "محاسب سوفت" واختيار النموذج الأنسب
## 📱 تحليل التطبيق
**نوع التطبيق:** نظام محاسبي عربي للهواتف الذكية (Android)
### الوظائف الرئيسية الظاهرة:
1. **قبض/صرف** - إدارة المقبوضات والمدفوعات
2. **المبيعات** - تسجيل عمليات البيع
3. **المشتريات** - تسجيل عمليات الشراء
4. **الحسابات** - إدارة الحسابات
5. **عمليات مخزنية** - إدارة المخزون
6. **قيود وحسابات** - القيود المحاسبية
7. **أصناف** - إدارة المنتجات
8. **العملات** - دعم متعدد العملات
9. **التقارير** - تقارير محاسبية
10. **المخازن** - دعم مخازن متعددة (المخزن الرئيسي)

### نقاط القوة:
✅ واجهة عربية بالكامل  
✅ تصميم بسيط وأيقونات واضحة  
✅ دعم العملات المتعددة  
✅ شمولية للوظائف المحاسبية الأساسية  
✅ وجود قناة يوتيوب للشرح  

### نقاط الضعف المحتملة:
⚠️ التصميم تقليدي ويحتاج تحديث  
⚠️ لا يظهر دعم للفوترة الإلكترونية  
⚠️ غير واضح إن كان يدعم المزامنة السحابية  
⚠️ لا يظهر دعم متعدد المستخدمين

## 💡 هل تريد المزيد؟

أخبرني ما الذي تريد تحليله بالتحديد:
- 🔍 تحليل تنافسي مع تطبيقات مشابهة؟
- 🎨 اقتراحات لتحسين الواجهة؟
- ⚙️ تحليل الوظائف وما ينقصها؟
- 📊 خطة تطوير وتحديث؟
- 💰 دراسة جدوى تجارية؟

اشرحلي هذه التبيبات ماهي الميزتها ولمذا نحتجها
في الصورة يظهر قسم "عمليات مخزنية"، وهو مسؤول عن إدارة حركة البضائع داخل المخازن. إليك شرح كل تبويب ولماذا نحتاجه:

1. صرف مخزني
- الوظيفة: إخراج أصناف من المخزن.
- يستخدم عند:
  - بيع بضاعة للعميل.
  - استهلاك مواد داخل الشركة.
  - إتلاف أو فقدان أصناف.
- الفائدة:
  - تقليل رصيد المخزون.
  - معرفة الكميات الخارجة بدقة.

مثال:
لديك 100 كرتون ماء، صرفت 20 كرتون لمندوب المبيعات، يصبح الرصيد 80.

2. توريد مخزني
- الوظيفة: إدخال أصناف إلى المخزن.
- يستخدم عند:
  - شراء بضاعة من المورد.
  - استرجاع بضاعة من عميل.
  - إضافة مخزون أول المدة.
- الفائدة:
  - زيادة رصيد المخزون.
  - تتبع الكميات الداخلة.

مثال:
اشتريت 50 جهازاً من المورد، يتم تسجيل توريد مخزني فتزداد الكمية.

3. تحويل مخزني
- الوظيفة: نقل أصناف من مخزن إلى مخزن آخر.
- يستخدم عندما يوجد أكثر من مخزن أو فرع.
- الفائدة:
  - المحافظة على دقة أرصدة كل مخزن.
  - معرفة مصدر ووجهة البضاعة.

مثال:
نقل 30 قطعة من المخزن الرئيسي إلى فرع الرياض.

4. تسوية مخزنية
- الوظيفة: تعديل الكميات لتطابق الواقع.
- يستخدم عند وجود فرق بين النظام والجرد الفعلي.
- الفائدة:
  - تصحيح الأخطاء.
  - معالجة الفقد أو الزيادة.

مثال:
النظام يظهر 100 قطعة لكن الجرد الفعلي 98 قطعة، يتم عمل تسوية بنقص 2 قطعة.

5. إضافة مخزن
- الوظيفة: إنشاء مخزن جديد داخل النظام.
- يستخدم عند:
  - فتح فرع جديد.
  - تخصيص مخزن لفئة معينة من الأصناف.
- الفائدة:
  - فصل المخزون حسب الموقع أو النشاط.

مثال:
إنشاء مخزن "المستودع الرئيسي" ومخزن "فرع جدة".

6. جرد مخزني
- الوظيفة: عدّ الأصناف الموجودة فعلياً في المخزن.
- يستخدم بشكل دوري (شهري أو سنوي).
- الفائدة:
  - اكتشاف النقص أو الزيادة.
  - التحقق من دقة البيانات.

مثال:
يتم عد جميع المنتجات ومقارنتها بأرصدة النظام.

كيف ترتبط هذه العمليات ببعضها؟
- التوريد ← يزيد المخزون.
- الصرف ← ينقص المخزون.
- التحويل ← ينقص مخزون ويزيد مخزوناً آخر.
- الجرد ← يفحص صحة الرصيد.
- التسوية ← تصحح الفروقات الناتجة من الجرد.
- إضافة مخزن ← تتيح إنشاء أماكن تخزين جديدة.

إذا كنت تريد بناء نظام محاسبي مشابه، فأعتبر "التوريد والصرف والتحويل والتسوية والجرد" هي العمليات الأساسية التي لا يمكن الاستغناء عنها في أي نظام مخزون احترافي.

هذا القسم يسمى "قيود وحسابات"، وهو قلب النظام المحاسبي، لأن جميع العمليات المالية تنتهي بقيود محاسبية تؤثر على الحسابات. إليك شرح كل تبويب:

1. قيد يومي
- الوظيفة: تسجيل العمليات المحاسبية يدوياً.
- يتكون من:
  - حساب مدين.
  - حساب دائن.
  - مبلغ.
  - وصف العملية.
- الفائدة:
  - تسجيل أي عملية لا تتم تلقائياً من المبيعات أو المشتريات.

مثال:
دفعت إيجار المحل 1000 ريال.
- مدين: مصروف الإيجار 1000
- دائن: الصندوق 1000

2. قيد افتتاحي
- الوظيفة: إدخال أرصدة بداية الفترة المالية.
- يستخدم عند:
  - بدء استخدام النظام لأول مرة.
  - بداية سنة مالية جديدة.
- الفائدة:
  - نقل الأرصدة السابقة إلى النظام.

مثال:
الصندوق يحتوي فعلياً على 5000 ريال عند بدء العمل بالنظام، يتم تسجيلها كقيد افتتاحي.

3. إضافة حساب
- الوظيفة: إنشاء حساب جديد في النظام.
- أمثلة:
  - صندوق.
  - بنك.
  - عميل.
  - مورد.
  - مصروفات.
  - إيرادات.
- الفائدة:
  - تنظيم جميع المعاملات المالية تحت حسابات محددة.

مثال:
إضافة حساب باسم "مصروف الإنترنت".

4. حركة الصندوق
- الوظيفة: عرض جميع العمليات النقدية.
- يظهر:
  - المقبوضات.
  - المدفوعات.
  - الرصيد الحالي.
- الفائدة:
  - متابعة النقدية لحظة بلحظة.

مثال:
معرفة كم دخل الصندوق اليوم وكم خرج منه.

5. دليل الحسابات
- الوظيفة: عرض شجرة الحسابات كاملة.
- يشمل:
  - الأصول.
  - الخصوم.
  - الإيرادات.
  - المصروفات.
  - حقوق الملكية.
- الفائدة:
  - تنظيم النظام المحاسبي بالكامل.

مثال:

الأصول
├─ الصندوق
├─ البنك
└─ العملاء

المصروفات
├─ الإيجار
└─ الرواتب

6. إقفال سنوي
- الوظيفة: إغلاق السنة المالية وفتح سنة جديدة.
- يقوم عادةً بـ:
  - تصفير الإيرادات والمصروفات.
  - ترحيل الأرباح أو الخسائر.
  - نقل الأرصدة المستمرة للسنة الجديدة.
- الفائدة:
  - استخراج نتائج السنة بدقة.
  - بدء سنة مالية جديدة بشكل صحيح.

مثال:
عند نهاية 2025 يتم إقفال الحسابات وبدء سنة 2026.

كيف ترتبط هذه التبويبات؟
- إضافة حساب ← إنشاء الحسابات.
- دليل الحسابات ← عرض وتنظيم الحسابات.
- قيد افتتاحي ← إدخال أرصدة البداية.
- قيد يومي ← تسجيل العمليات اليومية.
- حركة الصندوق ← متابعة النقدية.
- إقفال سنوي ← إنهاء الدورة المحاسبية.

إذا كنت تطور نظاماً محاسبياً، فإن "دليل الحسابات + القيود اليومية + القيود الافتتاحية + الإقفال السنوي" هي أساس المحاسبة المزدوجة ولا يمكن الاستغناء عنها في أي نظام احترافي.

هذا القسم هو "أصناف"، وهو المسؤول عن إدارة المنتجات والبضائع التي يتم بيعها أو شراؤها. يعتبر من أهم أجزاء أي نظام مبيعات ومخزون.

1. الأصناف
- الوظيفة: إنشاء وإدارة المنتجات.
- يحتوي عادة على:
  - اسم الصنف.
  - كود الصنف.
  - سعر الشراء.
  - سعر البيع.
  - الكمية المتاحة.
  - التصنيف.
- الفائدة:
  - توحيد بيانات المنتجات.
  - تسهيل عمليات البيع والشراء والجرد.

مثال:
صنف: لابتوب HP
الكود: HP001
سعر الشراء: 500$
سعر البيع: 650$

2. أسعار البيع
- الوظيفة: إدارة أسعار المنتجات.
- قد يدعم:
  - سعر الجملة.
  - سعر التجزئة.
  - سعر خاص للموزعين.
  - العروض والخصومات.
- الفائدة:
  - بيع نفس المنتج بأسعار مختلفة حسب العميل.

مثال:
منتج واحد:
- التجزئة: 100 ريال
- الجملة: 90 ريال
- موزع: 85 ريال

3. وحدات الصنف
- الوظيفة: تعريف وحدات القياس الخاصة بالمنتج.
- أمثلة:
  - قطعة.
  - كرتون.
  - صندوق.
  - كيلو.
  - متر.
  - لتر.
- الفائدة:
  - البيع والشراء بأكثر من وحدة.

مثال:
مياه معدنية:
- الوحدة الصغرى: زجاجة
- الوحدة الكبرى: كرتون
- 1 كرتون = 24 زجاجة

يمكن بيع زجاجة واحدة أو كرتون كامل.

4. فاتورة عرض سعر
- الوظيفة: إنشاء عرض سعر للعميل قبل البيع.
- لا تؤثر على المخزون أو الحسابات غالباً.
- الفائدة:
  - إعطاء العميل عرضاً رسمياً قبل اتخاذ قرار الشراء.

مثال:
عميل طلب معرفة تكلفة:
- 10 أجهزة كمبيوتر
- 5 طابعات

يتم إصدار عرض سعر بقيمة إجمالية دون تسجيل عملية بيع فعلية.

5. طلب شراء
- الوظيفة: طلب داخلي لشراء أصناف من الموردين.
- يسبق عملية الشراء الفعلية.
- الفائدة:
  - تنظيم احتياجات المخزون.
  - الموافقة على الطلبات قبل الشراء.

مثال:
المخزون وصل إلى حد منخفض:
- 100 علبة عصير
- 50 كرتون مياه

يتم إنشاء طلب شراء ثم تحويله لاحقاً إلى فاتورة مشتريات.

تسلسل العمل الطبيعي:
1. إنشاء الأصناف.
2. تحديد وحدات الصنف.
3. تحديد أسعار البيع.
4. إصدار عرض سعر للعميل (اختياري).
5. إنشاء فاتورة بيع.
6. عند نقص المخزون يتم إنشاء طلب شراء.
7. تحويل طلب الشراء إلى فاتورة مشتريات.
8. دخول البضاعة للمخزن.

إذا كنت تبني نظاماً محاسبياً جديداً، فالعناصر الأساسية هي:
- الأصناف.
- وحدات الصنف.
- أسعار البيع.

أما "عرض السعر" و"طلب الشراء" فهي ميزات متقدمة تجعل النظام أكثر احترافية وتناسب الشركات والمتاجر المتوسطة والكبيرة.

هذا القسم هو "العملات"، ويستخدم عندما تتعامل الشركة أو المتجر بأكثر من عملة واحدة. وهو مهم جداً للتجارة الدولية أو الشركات التي تتعامل مع موردين وعملاء من دول مختلفة.

1. إضافة عملة
- الوظيفة: إنشاء عملة جديدة داخل النظام.
- أمثلة:
  - ريال سعودي (SAR)
  - دولار أمريكي (USD)
  - يورو (EUR)
  - دينار كويتي (KWD)

الفائدة:
- تسجيل المبيعات والمشتريات بالعملة المناسبة.
- استخراج التقارير بعملات مختلفة.

مثال:
إذا كنت تشتري بالدولار وتبيع بالريال، تحتاج إلى إضافة العملتين للنظام.

2. سعر العملات
- الوظيفة: تحديد سعر الصرف بين العملات.
- الفائدة:
  - تحويل القيم من عملة إلى أخرى.
  - حساب الأرباح والخسائر الناتجة عن تغير أسعار الصرف.
  - إعداد التقارير المالية بعملة موحدة.

مثال:
1 دولار = 3.75 ريال

إذا اشتريت بضاعة بـ 100 دولار:
- النظام يحسبها = 375 ريال.

في الأنظمة المتقدمة يمكن تحديث السعر يومياً أو تلقائياً.

3. سقف الحساب
- الوظيفة: تحديد حد ائتماني للعميل أو المورد أو الحساب.
- المقصود غالباً:
  - الحد الأقصى للمبلغ المسموح به كدين.
  - أو الحد المسموح للحساب قبل التنبيه.

الفائدة:
- منع تجاوز العملاء للحد الائتماني.
- تقليل مخاطر الديون المتعثرة.

مثال:
عميل لديه سقف ائتماني 10,000 ريال.
إذا أصبحت مديونيته 9,500 ريال:
- النظام يحذرك.
إذا تجاوز 10,000 ريال:
- يمكن منع البيع الآجل له.

لماذا يوجد "سقف الحساب" داخل قسم العملات؟
من المحتمل أن مطور النظام وضعه هنا لأسباب تنظيمية، لكنه محاسبياً غالباً ينتمي إلى قسم العملاء أو الحسابات أكثر من قسم العملات.

كيف تعمل هذه العناصر معاً؟
- إضافة عملة → إنشاء العملات المستخدمة.
- سعر العملات → تحديد أسعار الصرف.
- سقف الحساب → ضبط حدود الائتمان للحسابات.

مثال عملي:
- تضيف الدولار والريال.
- تحدد أن 1 دولار = 3.75 ريال.
- تنشئ عميلاً بسقف ائتماني 20,000 ريال.
- تبيع له بالدولار.
- النظام يحول القيمة للريال ويتابع مديونيته مقارنة بالسقف المحدد.

هذه الميزات تصبح ضرورية إذا كان النشاط:
- استيراد وتصدير.
- تجارة إلكترونية دولية.
- التعامل مع موردين أجانب.
- شركات لديها فروع في أكثر من دولة.

أما إذا كان النشاط محلياً وبعملة واحدة فقط، فقد تحتاج فقط إلى العملة الأساسية ويمكن تجاهل معظم إعدادات العملات.

هذا قسم "التقارير"، وهو أهم جزء للإدارة واتخاذ القرار، لأن جميع البيانات التي تم إدخالها في المبيعات والمشتريات والمخزون والحسابات تتحول هنا إلى معلومات مفيدة.

شرح التقارير الظاهرة:

1. حركة الأصناف
- يعرض جميع الحركات التي تمت على كل صنف.
- يوضح:
  - الكميات الداخلة.
  - الكميات الخارجة.
  - الرصيد الحالي.
- يفيد في تتبع منتج معين.

مثال:
منتج دخل 100 قطعة وباع 70 قطعة، المتبقي 30 قطعة.

2. ميزان المراجعة
- تقرير محاسبي أساسي.
- يعرض جميع الحسابات وأرصدتها المدينة والدائنة.
- يستخدم للتأكد من توازن الحسابات.

الفائدة:
- اكتشاف الأخطاء المحاسبية قبل إعداد القوائم المالية.

3. قائمة الدخل
- تعرض:
  - الإيرادات.
  - المصروفات.
  - صافي الربح أو الخسارة.
- توضح أداء النشاط خلال فترة معينة.

مثال:
المبيعات = 100,000
المصروفات = 70,000
الربح = 30,000

4. المركز المالي
- يسمى أحياناً الميزانية العمومية.
- يعرض:
  - الأصول.
  - الخصوم.
  - حقوق الملكية.

الفائدة:
- معرفة الوضع المالي الحقيقي للشركة.

مثال:
الأصول = 500,000
الالتزامات = 200,000
حقوق الملكية = 300,000

5. تقارير أخرى
تحتوي على تقارير تفصيلية إضافية كما يظهر في الصورة الأولى:

• المخزون المتبقي
- يعرض الكميات الحالية لكل الأصناف.
- يساعد في معرفة النواقص والزيادات.

• أرصدة الحسابات
- يوضح رصيد كل حساب.
- مثل العملاء والموردين والصندوق والبنك.

• أرباح الأصناف
- يبين ربح كل منتج على حدة.

مثال:
سعر الشراء 50
سعر البيع 80
الربح 30

• العمليات اليومية
- جميع العمليات المنفذة خلال اليوم.

• القيود اليومية
- جميع القيود المحاسبية المسجلة.

• حركة الصندوق
- الإيرادات والمدفوعات النقدية.

• حركة الحسابات
- كشف حساب تفصيلي لأي حساب.

مثال:
كشف حساب عميل أو مورد.

• إجمالي التصنيفات
- المبيعات أو المشتريات حسب التصنيف.

مثال:
إلكترونيات = 50%
مواد غذائية = 30%
ملابس = 20%

• فوارق أسعار العملات
- الأرباح أو الخسائر الناتجة عن تغير سعر الصرف.

• رأس المال العامل
- يقيس قدرة الشركة على الوفاء بالتزاماتها قصيرة الأجل.

المعادلة:
الأصول المتداولة - الخصوم المتداولة

• الربح على مستوى العميل
- يحدد أكثر العملاء تحقيقاً للربح.

• المبيعات حسب الصنف
- أكثر المنتجات مبيعاً.

• المشتريات حسب الصنف
- أكثر المنتجات شراءً من الموردين.

• تفاصيل حركة الأصناف
- سجل كامل لكل حركة تمت على الصنف.

• الأصناف حسب الانتهاء
- الأصناف القريبة من انتهاء الصلاحية أو المنتهية.

مهم جداً:
إذا كنت تريد إنشاء نظام محاسبي حديث، فالتقارير الأساسية التي لا يجب الاستغناء عنها هي:
- قائمة الدخل.
- المركز المالي.
- ميزان المراجعة.
- حركة الأصناف.
- المخزون المتبقي.
- أرصدة الحسابات.
- حركة الصندوق.

أما باقي التقارير فهي تقارير تحليلية تساعد الإدارة على اتخاذ قرارات أفضل وزيادة الأرباح.

انا ايد أضافة كل هذه الحسابات في نظامي كا seed data جميعها بالكامل
نعم، إذا كنت تريد إضافة الدليل المحاسبي كـ Seed Data في نظامك، فأفضل طريقة هي عدم تخزينه كنصوص فقط، بل كشجرة حسابات (Chart of Accounts Tree) مرتبطة بـ parent_id.

وهل العملات توثر في الحسابات
نعم، العملات تؤثر على الحسابات بشكل مباشر إذا كان النظام يدعم تعدد العملات، لكن ليس بالطريقة التي يتخيلها الكثير.

من الصور يظهر أن النظام يحتوي على:
- العملات (محلي، دولار، سعودي).
- أسعار العملات بتاريخ معين.
- عدد الحسابات المرتبطة بكل عملة.

وهذا يعني أن كل حساب يمكن أن يكون له عملة افتراضية.

مثال:

- الصندوق (عملة محلية)
- صندوق الدولار (USD)
- بنك الرياض (SAR)
- عميل أجنبي (USD)

عند تسجيل قيد:

إذا بعت للعميل:
- 100 دولار
- سعر الصرف = 700

يسجل النظام:

العملة الأصلية:
- 100 USD

العملة المحلية:
- 70,000

وبالتالي يصبح للحساب رصيدين:
- الرصيد بالعملة الأصلية.
- الرصيد المكافئ بالعملة المحلية.

مثال عملي:

تم شراء بضاعة بـ:

```text
100 USD
سعر الصرف 700
```

يسجل:

```text
مدين  البضاعة      70,000
دائن  المورد       70,000
```

مع الاحتفاظ بأن العملية الأصلية:

```text
100 USD
```

ثم بعد أسبوع أصبح السعر:

```text
1 USD = 750
```

هنا يظهر فرق عملة:

```text
100 × (750 - 700)
= 5,000
```

ويتم تسجيله في الحساب الموجود عندك في الدليل:

```text
إيرادات أخرى
 └─ فوارق بيع وشراء العملات
```

ولهذا السبب وجدت حساب "فوارق بيع وشراء العملات" في الدليل المحاسبي الذي أرسلته، لأنه مرتبط مباشرة بوحدة العملات.

أما إذا كان نظامك سيعمل بعملة واحدة فقط، فيمكن تبسيط الأمر:
- لا تربط الحسابات بالعملات.
- اجعل هناك عملة أساسية واحدة للنظام.
- لا تحتاج جدول أسعار العملات ولا حساب فروقات العملات.

لكن إذا كنت تريد نظاماً احترافياً مثل النظام الموجود في الصور، فمن الأفضل:
- كل حساب له عملة افتراضية.
- كل حركة تحتفظ بالعملة الأصلية وسعر الصرف.
- يوجد حساب لفروقات العملات.
- التقارير تعرض الرصيد بالعملة الأصلية والمحلية معاً.

وهذه تفاصيل إنشاء المنتجات مع تحديد عملة ملاحظة العمل الأساسية هي العملة المحلية اين كانت هي التي يتم التحول إلها وسعر التكلفة يكون بعملة المحلية والعملة المحلية هي عملة المحلية لمستخدم النظام اين كانت هاذه العملة وللعلم اقل شيى يكون لكل منتج وحدتين على الاقل والوحدة الصغرى وهي الوحدة الاساسية ووحدة اخرى فرعية اخرى على الاقل وهذا اقل شيء موجود في السوق وبهذه الطريقة يمكن للمستخد البيع بألي وحدة يرد والنظام يقوم بحتساب الكميات والاسعار وهذا يسهل ويسرع على المستخدم
الفكرة التي يطبقها هذا النظام جيدة جداً، وفي الحقيقة أقرب لما تستخدمه أنظمة ERP التجارية أكثر من أنظمة نقاط البيع البسيطة.

من الصور والملاحظات التي ذكرتها أستنتج أن تصميم المنتجات يجب أن يكون كالتالي:

1. العملة الأساسية للنظام (Base Currency)

يوجد عملة أساسية واحدة للنظام يحددها المستخدم عند إنشاء الشركة.

مثال:
- اليمني (YER)
- السعودي (SAR)
- الدولار (USD)

جميع التكاليف المحاسبية والمخزون تحفظ بهذه العملة.

مثال:

```text
العملة الأساسية = ريال يمني

تكلفة المنتج = 2000 ريال يمني
```

حتى لو تم البيع بالدولار أو الريال السعودي.

-----------------------------------

2. عملة سعر البيع

الصورة توضح أن سعر البيع يمكن تحديده بأي عملة:

مثال:

```text
بطاطس عمان

التكلفة:
2000 YER

أسعار البيع:
1 USD
5 SAR
3000 YER
```

وعند البيع:

إذا اختار العميل الدولار:

```text
1 USD
```

النظام يحولها تلقائياً إلى العملة الأساسية لأغراض المحاسبة.

مثال:

```text
1 USD = 700 YER

قيمة القيد = 700 YER
```

لكن الفاتورة تبقى بالدولار.

وهذا هو التصرف الصحيح محاسبياً.

3. الوحدات

أوافقك أن وجود وحدة واحدة فقط غير عملي.

الأفضل أن يكون:

```text
الوحدة الأساسية (إجبارية)
الوحدات الفرعية (واحدة أو أكثر)
```

مثال:

```text
منتج: مياه

الوحدة الأساسية:
حبة

الوحدات الفرعية:
كرتون = 24 حبة
بالتة = 48 كرتون
```

-----------------------------------

4. تخزين المخزون

لا يتم تخزين المخزون إلا بالوحدة الأساسية.

مثال:

```text
1 كرتون = 24 حبة

دخل للمخزن:
10 كراتين

المخزون الفعلي:
240 حبة
```

-----------------------------------

5. البيع بأي وحدة

مثال:

المستخدم يبيع:

```text
2 كرتون
```

النظام يحسب:

```text
2 × 24 = 48 حبة
```

ثم يخصم:

```text
المخزون = المخزون - 48
```

-----------------------------------

6. أسعار مستقلة لكل وحدة

مثال:

```text
حبة = 100 ريال

كرتون = 2200 ريال

بالتة = 100000 ريال
```

وليس بالضرورة أن يكون سعر الكرتون = 24 × سعر الحبة.

لأن الجملة غالباً فيها خصم.

الخلاصة

إذا كنت تريد نظاماً احترافياً وقابلاً للتوسع:

- عملة أساسية واحدة للشركة.
- حفظ التكاليف والمحاسبة بالعملة الأساسية فقط.
- السماح بأسعار بيع متعددة العملات.
- وحدة أساسية إجبارية.
- وحدة إضافية واحدة على الأقل.
- تخزين المخزون دائماً بالوحدة الأساسية.
- البيع والشراء بأي وحدة.
- أسعار مستقلة لكل وحدة ولكل عملة.
- تحويل تلقائي للعملة الأساسية عند إنشاء القيود المحاسبية.

هذا التصميم يغطي تقريباً كل ما رأيته في "محاسب سوفت" مع بنية أكثر مرونة وقابلة للتوسع مستقبلاً.

وهذه التفصيلية في في خفض تاريخ اسعار البيع وما فائدة هذه الميزة
هذه من الميزات المهمة جداً في الأنظمة المحاسبية والمخزنية الاحترافية، وتسمى:

- Price History (سجل الأسعار)
- Price Versioning (إصدارات الأسعار)
- Effective Pricing (الأسعار حسب تاريخ النفاذ)

من الصورة يظهر أن النظام لا يقوم بتعديل السعر القديم، بل يحتفظ بكل تغيير حصل على سعر المنتج مع تاريخه.

مثال من الصورة:

بطاطس عمان - حبة

```text
2026-05-31   100 YR
2026-05-31   150 YR
2026-06-01   1 USD
```

بطاطس عمان - كرتون

```text
2026-05-31   3000 YR
2026-06-01   300 YR
```

لماذا هذه الميزة مهمة؟

1. معرفة السعر الذي كان مستخدماً وقت البيع

مثال:

بعت يوم:

```text
2026-05-31
```

وكان السعر:

```text
100 ريال
```

بعد شهر غيرت السعر إلى:

```text
150 ريال
```

عند مراجعة فاتورة قديمة يجب أن يبقى السعر:

```text

```

كم أصبحت تكلفة المنتج الآن؟

الإجابة تعتمد على طريقة تقييم المخزون التي ستعتمدها في النظام.

1. المتوسط المرجح (Weighted Average) ← أنصح به

الأكثر استخداماً في الأنظمة التجارية.

الحساب:

```text
(100 × 50) + (50 × 70)
----------------------
100 + 50
```

النتيجة:

```text
8500 ÷ 150

= 56.67
```

تصبح تكلفة المنتج الحالية:

```text
56.67
```

والمخزون:

```text
150 حبة
```

عند البيع يتم استخدام:

```text
56.67
```

كتكلفة.

-----------------------------------

2. الوارد أولاً صادر أولاً (FIFO)

هذه الطريقة مناسبة أكثر للمواد الغذائية والأدوية.

تبقى كل دفعة مستقلة.

دفعة 1:

```text
100 حبة
تكلفة 50
انتهاء 2026-12-31
```

دفعة 2:

```text
50 حبة
تكلفة 70
انتهاء 2027-03-31
```

إذا بعت:

```text
20 حبة
```

يخصم من الدفعة الأولى.

يبقى:

```text
80 حبة × 50
50 حبة × 70
```

إذا بعت:

```text
100 حبة
```

سيستهلك:

```text
80 من الدفعة الأولى
20 من الدفعة الثانية
```

وهو الأفضل للمواد الغذائية.

في الأنظمة الاحترافية يتم التعامل مع ذلك بإحدى الطريقتين:

1. تفعيل تتبع الانتهاء حسب المنتج

في بطاقة المنتج يوجد حقل:

```text
هل المنتج له تاريخ انتهاء؟
✓ نعم
✗ لا
```

أمثلة:

لها انتهاء:
- الأدوية
- الأغذية
- العصائر
- مستحضرات التجميل

ليس لها انتهاء:
- الحديد
- الإسمنت
- الأدوات الكهربائية
- قطع الغيار
- الأجهزة الإلكترونية (غالباً)

إذا كانت الإجابة "لا":

```text
expiry_date = NULL
```

ولا يطلب النظام تاريخ انتهاء عند الشراء أو الجرد.

2. استمرار استخدام الدفعات بدون انتهاء

```text
cost=50
expiry=NULL
```

```text

cost=70
expiry=NULL
```


```text

```

إذا أردنا بناء نظام قوي وقابل للتوسع لعشر سنوات قادمة، فأنا أقترح أن نبدأ الآن بتحليل:

"شاشة إعداد الشركة"

تحليلاً كاملاً:

- هدف الشاشة.
- جميع الحقول.
- قواعد التحقق.
- تجربة المستخدم.
- الجداول الناتجة.
- العلاقات.
- Seed Data المطلوبة.

ثم بعد الانتهاء منها ننتقل للشاشة التالية خطوة بخطوة حتى نصل إلى Database Schema النهائي الكامل.

ويوجد مع النظام عملتين أفتراضيتين المحلية والدولار مع النظام والعملة الاساسية هي العملة المحلية   "اولا اعدادت النظام مثل الشركة وغيرها والعملة تكون في شاشة الاعداد لنظام ويكون هناك قيم أفتراضية مثل "الحنيني للأنظمة الالكترونية
ممتاز، هذه نقطة مهمة لأنها تؤثر على التصميم من البداية.

أنا أنصح ألا نفكر في "إعداد الشركة" فقط، بل في "إعداد النظام لأول تشغيل" (First Run Setup).

وبناءً على ما شرحته عن النظام المستهدف (مشابه لمحاسب سوفت لكن بتصميم احترافي وقابل للتوسع)، فأرى أن النسخة الأولى يجب أن تعمل كالتالي:

الهدف

عند تثبيت النظام لأول مرة:

- لا يكون هناك أي بيانات.
- يتم إنشاء شركة افتراضية.
- يتم إنشاء العملات الافتراضية.
- يتم إنشاء الدليل المحاسبي الافتراضي.
- يتم إنشاء مخزن افتراضي.
- يتم إنشاء صندوق افتراضي.
- يتم إنشاء مستخدم مدير النظام.

بدون أن يضطر المستخدم لإنشاء كل شيء يدوياً.

أمر مهم بخصوص العملات

أنا لا أنصح باستخدام اسم:

```text
محلي
```

كعملة.

بل نخزن العملة الحقيقية.

مثال:

بدلاً من:

```text
محلي
```

نخزن:

```text
YER
```

لكن في الواجهة يمكن عرض:

```text
العملة المحلية
```

للمستخدم.

لأن التقارير والفواتير والمحاسبة تحتاج الرمز الحقيقي.

-----------------------------------

من وجهة نظري، أول 5 جداول يجب تصميمها قبل أي شيء هي:

```text
Companies
```

```text
Currencies
```

```text
CurrencyRates
```

```text
FiscalYears
```

```text
SystemSettings
```

وبعد الانتهاء منها ننتقل مباشرة إلى:

"دليل الحسابات (Chart of Accounts)"

لأنه هو الأساس الحقيقي الذي ستبنى عليه:
- الصناديق.
- العملاء.
- الموردون.
- المبيعات.
- المشتريات.
- القيود.
- التقارير المالية.

لا هذه الابيانات الافتراضية تكون مع النظام عند عمل تشغيل Migations in C# as seed data
ممتاز، الآن فهمت قصدك بشكل أدق.

أنت لا تريد Wizard إعداد أول تشغيل.

بل تريد أن تأتي بعض البيانات جاهزة مع النظام كـ Seed Data داخل EF Core Migrations.

وهذا هو الأسلوب الذي أفضله أيضاً للنسخة الأولى لأنه يجعل النظام يعمل مباشرة بعد التثبيت.

في هذه الحالة نفصل بين:

1. Seed Data ثابتة مع النظام.
2. بيانات الشركة التي يدخلها المستخدم.
