في تصميمك الحالي:

```text
FIFO
+
InventoryBatches
+
QuantityRemaining
+
UnitCost
```

فالإجابة هي:

❌ لا أنصح بهذا الإعداد في V1.

السبب أن طريقة احتساب التكلفة عندك أصبحت قراراً معمارياً في النظام وليست إعداداً للمستخدم.

أنت بالفعل قررت:

```text
FIFO
```

لذلك لا معنى لوجود:

```text
CostingMethod
```

بقيم:

```text
1=WeightedAverage

2=LastPurchasePrice

3=SupplierPrice
```

لأن هذا يعني أن النظام يدعم 3 محركات تكلفة مختلفة، بينما قاعدة البيانات والقيود التي صممناها مبنية على FIFO.

━━━━━━━━━━━━━━━━━━━━
ما المشكلة لو أبقيته؟
━━━━━━━━━━━━━━━━━━━━

سيظهر للمستخدم:

```text
طريقة التكلفة

○ متوسط مرجح
○ آخر سعر شراء
○ سعر المورد
```

لكن فعلياً:

```text
النظام يعمل FIFO
```

وهذا يسبب تضارباً وخطأً محاسبياً.

━━━━━━━━━━━━━━━━━━━━
ماذا أفعل إذا أردت دعمه مستقبلاً؟
━━━━━━━━━━━━━━━━━━━━

إذا كنت تتوقع مستقبلاً دعم:

```text
FIFO
Weighted Average
```

فيمكنك إضافة الإعداد لاحقاً.

مثال:

```text
CostingMethod

1 FIFO
2 WeightedAverage
```

لكن لا تضفه الآن إذا لم تنوِ تنفيذ الطريقتين فعلياً.

━━━━━━━━━━━━━━━━━━━━
ما الإعداد الصحيح حالياً؟
━━━━━━━━━━━━━━━━━━━━

بدلاً من:

```text
CostingMethod
```

يمكنك إضافة إعدادات أكثر فائدة لنظامك مثل:

```text
AllowNegativeStock
```

السماح بالمخزون السالب.

━━━━━━━━━━━━━━━━━━━━

```text
AllowSalesBelowCost
```

السماح بالبيع بأقل من التكلفة.

━━━━━━━━━━━━━━━━━━━━

```text
AutoSelectBatchByExpiry
```

اختيار الدفعة تلقائياً حسب FEFO.

━━━━━━━━━━━━━━━━━━━━

```text
DefaultTaxId
```

الضريبة الافتراضية.

━━━━━━━━━━━━━━━━━━━━

```text
ExpiryAlertDays
```

عدد أيام التنبيه قبل الانتهاء.

━━━━━━━━━━━━━━━━━━━━
الخلاصة
━━━━━━━━━━━━━━━━━━━━

بما أنك اعتمدت:

```text
FIFO
```

فاحذف:

```text
CostingMethod
```

من V1.

ولا تضف إعداداً يسمح بتغيير طريقة التكلفة إلا إذا كنت ستدعم فعلياً أكثر من محرك تكلفة داخل النظام. هذا سيجعل التصميم أبسط وأكثر اتساقاً محاسبياً.
