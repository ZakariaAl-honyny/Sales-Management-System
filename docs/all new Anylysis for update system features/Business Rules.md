# تحليل قواعد الأعمال — Business Rules Analysis

## مقدمة
توثيق متى يسمح بالحذف والتعديل والإلغاء والترحيل لكل كيان في النظام.

---

## 1. دورة حياة المستندات (Document Lifecycle)

جميع المستندات المالية تتبع دورة حياة موحدة من 3 حالات:
```
مسودة (Draft) → مرحّل (Posted) → ملغي (Cancelled)
```

| الحالة | الرقم | المعنى |
|--------|-------|--------|
| Draft | 1 | مسودة — يمكن تعديلها |
| Posted | 2 | مرحّلة — تم ترحيلها وتأثيرها على المخزون/الحسابات |
| Cancelled | 3 | ملغية — حالة نهائية، لا يمكن التراجع |

**قواعد عامة:**
- ✅ Draft → Posted (ترحيل)
- ✅ Draft → Cancelled (إلغاء بدون تأثير)
- ✅ Posted → Cancelled (إلغاء مع عكس التأثير)
- ❌ Posted → Draft (ممنوع نهائياً)
- ❌ Cancelled → أي حالة (حالة نهائية)

## 2. فواتير البيع (SalesInvoice)

| العملية | مسموح؟ | الشرط | التأثير |
|---------|--------|-------|---------|
| **إنشاء** | ✅ | أصناف صحيحة | مسودة |
| **تعديل** | ✅ فقط في Draft | Status == Draft | إضافة/إزالة أصناف |
| **ترحيل (Post)** | ✅ | Draft + وجود أصناف | يحدث المخزون + الرصيد + قيود محاسبية |
| **إلغاء (Cancel)** | ✅ Draft/Posted | PaidAmount == 0 | عكس المخزون + الرصيد + قيود |
| **حذف نهائي** | ❌ ممنوع | — | استخدم الإلغاء بدلاً من الحذف |

**القيود:**
- `PaidAmount <= TotalAmount` — المبلغ المدفوع لا يتجاوز الإجمالي
- `DueAmount = TotalAmount - PaidAmount` — محسوب دائماً
- لا يمكن إلغاء فاتورة مدفوعة (يجب عمل مردود أو تحصيل أولاً)

| **حذف مسودة** | ✅ (حذف مباشر) | Status == Draft | لا تأثير مالي — يمكن حذفها مباشرة |

**ملاحظة مهمة:** قيد اليومية لا يسمح بـ Draft→Cancelled لأنه ليس له تأثير مالي في حالة المسودة. المسودة إما تُرحّل أو تُحذف مباشرة.

## 15. قواعد عامة للحذف

| القاعدة | الشرح |
|---------|-------|
| **لا حذف مادي للمستندات المالية** | كل المستندات (فواتير، سندات، قيود) تستخدم دورة الحالة (Draft→Posted→Cancelled) |
| **الحذف الناعم للكيانات الرئيسية** | المنتجات، العملاء، الموردين، المستخدمين تستخدم IsActive = false |
| **فحص المراجع قبل الحذف النهائي** | التأكد من عدم وجود معاملات مرتبطة قبل الحذف النهائي |
| **حماية حسابات النظام** | الحسابات النظامية (IsSystem/IsSystemAccount) لا يمكن حذفها أو تعديلها |
| **حماية المستخدمين** | لا يمكن حذف مستخدم نهائياً — فقط إلغاء تنشيط (لأن الفواتير مرتبطة به) |

## 16. مصفوفة الحذف حسب استراتيجية DeleteStrategy

| الاستراتيجية | الرقم | المعنى |
|-------------|-------|--------|
| Cancel | 0 | إلغاء العملية (للمستخدم: يتراجع عن الحذف) |
| Deactivate | 1 | حذف ناعم — تعيين IsActive = false |
| Permanent | 2 | حذف نهائي — إزالة مادية من قاعدة البيانات |

**ملاحظة:** `DeleteStrategy` معرّف في الكود لكن الخدمات تستخدم دوال منفصلة (`DeleteAsync` + `PermanentDeleteAsync`) بدلاً من استراتيجية موحدة. هذا مقبول حالياً لكن يمكن توحيده مستقبلاً.

## 17. قواعد التحقق من الصلاحية (Guards) — نموذج موحد

جميع الكيانات تتبع نفس النمط:

```csharp
if (condition)
    throw new DomainException("الرسالة بالعربية");
```

| نوع التحقق | أمثلة |
|-----------|-------|
| **حقول إلزامية** | `IsNullOrWhiteSpace(name)`, `supplierId <= 0` |
| **نطاق القيم** | `amount <= 0`, `quantity < 0` |
| **حالة المستند** | `Status != InvoiceStatus.Draft` للتعديل |
| **التكرار** | `Status == InvoiceStatus.Cancelled` للإلغاء المزدوج |
| **قواعد العمل** | `PaidAmount > TotalAmount`, `!IsBalanced()` |
| **المراجع** | `Items.Count == 0` قبل الترحيل |
| **الحصرية** | `sourceWarehouseId == destinationWarehouseId` للتحويل |
